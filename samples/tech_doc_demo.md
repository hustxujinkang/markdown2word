# 用户活跃度统计服务技术方案

作者:王五  ·  版本:v0.3  ·  状态:评审中

## 1. 背景与目标

### 1.1 业务背景

随着用户规模增长,业务方对"用户活跃度"指标的查询频率提升,现有方案存在以下问题:

- 实时查询直接打到主库,QPS 峰值时段拖慢业务表读取
- 统计口径分散在各业务系统,数据一致性差
- 历史回溯查询(超过 7 天)依赖临时跑批,响应慢且重复劳动

### 1.2 设计目标

设计并实现一个**统一的用户活跃度统计服务**,达成:

| 维度 | 目标 |
|------|------|
| 查询性能 | P99 响应 < 200ms |
| 数据时效 | 准实时(< 5 分钟延迟) |
| 历史回溯 | 支持任意时段查询 |
| 数据一致性 | 与业务库差异 < 0.1% |

### 2.2 核心组件

1. **数据采集层**:订阅业务库 binlog,经 Kafka 投递
2. **流处理层**:Flink 作业聚合用户行为,5 分钟一个窗口
3. **存储层**:ClickHouse 列存,按用户分区
4. **服务层**:Go 编写的查询服务,内置多级缓存

## 3. 关键技术决策

### 3.1 为什么选 ClickHouse 而不是 Druid

> 在数据量级 < 10 亿行、单点写入吞吐 < 50 MB/s 的场景下,ClickHouse 的查询性能优于 Druid 约 30%,且运维成本更低。Druid 的优势主要体现在更大规模(> 100 亿行)的查询并发场景,本场景用不到。

### 3.2 缓存策略

```go
// 三层缓存:本地 LRU → Redis → ClickHouse
func (s *Service) Query(uid int64, day string) (*Stats, error) {
    if v, ok := s.lru.Get(key(uid, day)); ok {
        return v.(*Stats), nil
    }
    if v, err := s.redis.Get(key(uid, day)); err == nil {
        return v, nil
    }
    return s.queryFromCH(uid, day)
}
```

本地 LRU 命中率预估 35%,Redis 命中率预估 50%,落到 ClickHouse 的请求约占 15%。

## 4. 容量与成本

### 4.1 数据量估算

- 用户基数:1500 万
- 日活:500 万
- 每用户每日产生事件:平均 40 条
- 单条事件大小:120 字节

日增数据量约 24 GB,年增 8.7 TB。按 ClickHouse 3 倍压缩比,实际存储约 2.9 TB。

## 5. 上线计划

1. **第 1-2 周**:基础架构搭建,数据通路打通
2. **第 3-4 周**:核心查询接口开发,完成单元测试
3. **第 5 周**:压测与调优,目标达成 P99 < 200ms
4. **第 6 周**:灰度上线,逐步切流至 100%

## 6. 风险与应对

主要风险点:Flink 作业稳定性。应对方案:
- 双流冗余部署
- 异常恢复时支持从 Kafka 任意位点回放
- 关键指标看板 + 告警机制
